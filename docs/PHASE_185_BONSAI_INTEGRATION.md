## Summary

Phase 185 turns STING Tools from a Revit-only plugin into the data-layer
spine of a multi-host BIM coordination platform. It does this by:

1. **Codifying the data layer** as an IFC4-native substrate (52 enums + 2
   psets + 2 IDS files + bSDD publication plan), with SHA-256 corporate
   locks and project-overlay extensibility.
2. **Providing a dual-language core** (`stingtools-core/python/` shipped;
   `stingtools-core/dotnet/` planned) that every host plugin and the
   Planscape server read from.
3. **Standing up the first non-Revit host plugin** as an extension to
   Bonsai (formerly BlenderBIM), positioned as a coexistence layer
   rather than a replacement.
4. **Federating across hosts** via a new Planscape Server IFC-ingest
   endpoint + the `ExternalElementMapping` cross-host identity table.

The end-state Phase 185 enables: an issue raised in Bonsai on an IFC
`GlobalId` surfaces in Revit on the matching `ElementId`, validated against
the same IDS specs, with a single audit-log chain.

### Architectural decisions

**D1 — Code namespacing is per-enum, not global.**
Same code (`Standard`, `None`, `XX`) appears in multiple enums with
different meanings. UIs must disambiguate by enum context.
Documented in `enums/_README.md § Code namespacing`.

**D2 — Three of 52 enums are project-scoped templates.**
`StingLocationCodes`, `StingZoneCodes`, `StingLevelCodes` ship empty
(sentinels only + placeholders) and projects fill via overlay at
`<project>/_BIM_COORD/enums/`. Reserved sentinels (`XX`, `*`, `SITE`,
`EXT`, `GF`, `MZ`, `RF`, `PR`, `ZZ`) cannot be redefined.

**D3 — SHA-256 corporate lock pins value-set semantics.**
Canonical JSON form excludes definitions / descriptions / governance
metadata — only `code`, `sentinel`, `since`, `deprecated`, `display_form`,
`replaced_by` participate in the hash. Description edits don't flip the
lock; semantic edits do.

**D4 — `display_form` attribute decouples internal code from rendered form.**
`StingConcreteGrades` uses `C30_37` as the XSD-compatible code, with
`display_form="C30/37"` for UIs. Pattern generalises to any standard
notation that doesn't fit `[A-Za-z0-9_\-\*]+`.

**D5 — Pset cross-entity rules sit alongside IDS specs, not inside them.**
IDS v1.0 cannot express "element.Loc equals containing-building.LocationCode".
The Pset XML carries `<CrossEntityValidationRules>` as a declarative
description; IDS files encode the parts IDS can express (presence,
pattern, enum membership); STING-side `SpatialChecker` closes the
equality gap by walking the IFC graph after IDS passes.

**D6 — bSDD publication is opt-in per enum.**
Healthcare enums (HTM/HBN/NCRP-derived) marked `ready` for STING
publication under their respective namespaces. STING-proprietary
taxonomies (`StingProductCodes`, `StingDrawingPurposes`) marked
`private`. Six enums marked `external_already` where buildingSMART /
RIBA / BCF already own the term set.

## The Python core — `stingtools-core/python/`

```
stingtools_core/
├── paths.py                find_shared_ifc() — env-var override or parent-walk
├── enums/                  EnumRegistry + Enum + EnumValue + SHA-256 verification + overlay merge
├── psets/                  PsetRegistry + Pset + PsetProperty + ValidationRule
├── tag_grammar/            Tag (8-segment dataclass) + TagGrammar (builder) + TagValidationResult
├── spatial/                SpatialChecker (cross-entity LOC/LVL/ZONE equality on ifcopenshell models)
├── ids/                    IdsRunner (ifctester wrapper, skip-if-missing)
└── planscape/              PlanscapeClient (HTTP/JWT) + AuditLog (SHA-256 chained JSONL)
```

### Architectural decisions

**D7 — Lazy optional dependencies.**
Every optional integration (`ifctester`, `ifcopenshell`, `httpx`,
`signalrcore`) is imported via `_try_import()` so the package loads
cleanly without them. CLI tooling, server-side ingest, and CI all
exercise the package without needing the full BIM stack.

**D8 — `paths.find_shared_ifc()` walks parents OR honours
`STINGTOOLS_SHARED_IFC` env var.**
The env-var form is the deployment story for Blender extensions where
the substrate is bundled under `_vendor/` rather than discoverable via
filesystem-walk.

**D9 — `Tag` is an immutable dataclass; `TagGrammar` is the stateful
wrapper.**
`Tag(...)` parses + renders without a registry; `TagGrammar(registry)`
gains validation + stage-required-segment logic. Cleanly separates the
data shape from the rules that interpret it.

**D10 — `AuditLog` uses identical SHA-256-chained JSONL format as the
C# AuditLog.**
Server-side verification can read entries from any host plugin without
language-specific deserialisation. Format spec in
`stingtools_core.planscape.audit_log.AuditLog`.

### Public API surface


### Architectural decisions

**D11 — `ExternalElementMapping` is composite-keyed on
(ProjectId, IfcGlobalId, Host, HostDocumentGuid).**
Same IFC GlobalId can appear in multiple federated documents (e.g. arch
RVT + structural RVT both export an IFC referencing the same wall);
each pairing gets its own row + its own `HostElementId`.

**D12 — `TaggedElement` unique constraints made filtered.**
Pre-Phase-185, `(ProjectId, RevitElementId)` was unique unconditional —
non-Revit hosts setting `RevitElementId = 0` would collide. Now:
- Unique `(ProjectId, RevitElementId) WHERE RevitElementId > 0` — preserves Revit-side semantics
- Unique `(ProjectId, UniqueId) WHERE UniqueId <> ''` — cross-host key
  (Revit `UniqueId` for Revit; IFC `GlobalId` for everything else)

**D13 — `POST /api/projects/{id}/ifc/data` accepts five host values.**
`revit | blender | archicad | tekla | headless`. `headless` is reserved
for server-side automation (e.g. nightly batch ingest jobs).
Validation is hard-coded; future hosts add a manifest constant rather
than free-form strings.

**D14 — `GET /api/projects/{id}/ifc/mappings?ifc_guid=...` is the
cross-host join endpoint.**
This is what makes "issue raised in Blender shows up in Revit" work —
the issue carries an IfcGlobalId; the Revit plugin queries this
endpoint to resolve back to a Revit `UniqueId`, then displays the issue
against the matching element.

## The Bonsai add-on — `stingtools-bonsai/`

### Architectural decisions

**D15 — Bonsai-coexistence, not Bonsai-replacement.**
Two add-ons share the same IFC file via `ifcopenshell.api.run()`.
Bonsai owns: IFC load/save, geometry, classifications, BCF, drawings.
STING owns: 8-segment tag grammar, IDS validation, Planscape federation,
audit log, healthcare validators (future). Documented in
`stingtools-bonsai/BONSAI_RELATIONSHIP.md`.

**D16 — `BonsaiBridge.add_pset()` is the only sanctioned write path.**
Routes every IFC mutation through `ifcopenshell.api.run("pset.add_pset", ...)`
so Bonsai's undo + UI refresh hook in. Standalone-mode fallback calls
the same API directly with reduced undo fidelity.

**D17 — Detection probes 3 module names.**
Bonsai has gone through two renamings: `blenderbim` → `bonsai` →
`bonsai_bim` (extensions form). `BonsaiBridge._probe()` tries each.

**D18 — Sidebar tab is "STING", not "BIM".**
Bonsai owns the "BIM" tab. STING gets its own. Users see both side by
side; STING never overrides Bonsai's UI surfaces.

**D19 — Vendored `stingtools_core` under `_vendor/` for built extensions.**
Dev mode: `__init__.py` injects `../stingtools-core/python/` onto
`sys.path`. Release mode: the build step copies the package under
`_vendor/`. Either way, no PyPI dependency the user has to install.

## Cross-host federation model

```
                        ┌──────────────────────────────┐
                        │     Planscape Server         │
                        │                              │
                        │  ┌────────────────────────┐  │
                        │  │ ExternalElementMapping │  │
                        │  │  (project, ifc_guid,   │  │
                        │  │   host, host_doc)      │  │
                        │  │   → host_element_id    │  │
                        │  └────────────────────────┘  │
                        │  ┌────────────────────────┐  │
                        │  │ TaggedElement          │  │
                        │  │ BimIssue               │  │
                        │  │ AuditLog               │  │
                        │  └────────────────────────┘  │
                        └──────┬─────────────┬─────────┘
                               │             │
                  POST /ifc/data  GET /mappings  + SignalR
                               │             │
        ┌──────────────────────┼─────────────┼──────────────────────┐
        │                      │             │                      │
   ┌────▼────┐            ┌────▼─────┐  ┌────▼─────┐           ┌────▼─────┐
   │  Revit  │            │  Bonsai  │  │ ArchiCAD │           │  Tekla   │
   │ plugin  │            │ + STING  │  │ + STING  │           │ + STING  │
   │ (today) │            │ (Day-1)  │  │ (future) │           │ (future) │
   └─────────┘            └──────────┘  └──────────┘           └──────────┘
```

The keystone is **IFC `GlobalId` as the cross-host stable identifier**.
Each host plugin pushes `(ifc_guid, host_element_id)` pairs through the
ingest endpoint; the server's mapping table is the lookup oracle.

## Decisions log — key trade-offs

| Decision | Alternative considered | Why this won |
|---|---|---|
| IFC4 Reference View as the MVD | Coordination View 2.0 | Reference View is the Bonsai default + lightweight enough for federation |
| Pset XML in STING-custom format | buildingSMART PSD format direct | STING-custom carries governance + lock + cross-entity rules PSD can't; converter projects to PSD when needed |
| SHA-256 over canonical JSON, not raw XML | Hash the XML file | XML whitespace + comment edits would flip the lock; canonical JSON is semantic |
| 3 enums project-scoped, 49 corporate-locked | All corporate | Spatial codes are inherently per-project; locking them would require schema-release per new building |
| `display_form` attribute for non-pattern codes | Allow `/` in XSD pattern | XSD compatibility with stricter validators; `_` → `/` substitution is documented |
| Bonsai coexistence, not replacement | Fork Bonsai or build standalone | Bonsai community + IFC mass; STING adds ISO 19650 layer they don't have |
| Composite key `(project, ifc_guid, host, host_doc)` | Just `(project, ifc_guid)` | Federated models share GlobalIds across documents |
| EF filtered unique on TaggedElement | Drop unique entirely | Preserves Revit-side ingest semantics; non-Revit hosts use UniqueId path |
| Vendored `stingtools_core` under `_vendor/` | PyPI dependency in extension manifest | Blender extension manifests don't reliably support pip-installable deps yet |
